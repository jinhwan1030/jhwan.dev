## 왜 만들었나

아이를 키우다 보면 매일 아침 같은 고민을 하게 된다. 오늘 나가도 될까? 미세먼지는? 자외선은? 너무 덥진 않을까?

날씨 앱을 열고, 미세먼지 앱을 따로 열고, 자외선 지수를 검색하고... 이 과정이 매번 번거로웠다. 그래서 그냥 만들기로 했다. 장소를 검색하면 외출 적합도를 0~100점으로 보여주는 서비스.

아이날씨는 단순히 날씨 정보를 보여주는 앱이 아니라, 실제 외출 결정을 빠르게 돕는 작은 제품을 목표로 만들었다. 지금은 직접 운영하면서 데이터 신뢰성, 모바일 사용성, 배포 자동화까지 계속 다듬고 있다.

## 무엇을 만들었나

아이날씨는 위치를 검색하면 현재 또는 선택한 미래 시간 기준으로 아이와 외출하기 좋은지를 점수로 보여주는 모바일 웹앱이다.

- 장소명 검색
- 현재, +1h, +3h, +6h, +12h 및 직접 시간 선택
- 기온, 강수, 자외선, 미세먼지 점수화
- 기상청과 에어코리아 데이터 출처 표시
- PWA 아이콘과 홈 화면 추가 지원
- 사용자용 업데이트 내역과 피드백 경로 제공

## 기술 스택

```
Frontend: Vanilla JS + Tailwind CDN
Backend: FastAPI (Python)
HTTP Client: httpx AsyncClient
Cache: In-memory LRU + Single-flight Lock
Infra: Docker + Docker Compose + Nginx
CI/CD: GitHub Actions + Docker Hub multi-arch image
Runtime: 라즈베리파이 4B
```

데이터는 세 곳에서 가져온다.

- **기상청 실황/초단기예보/단기예보 API** — 기온, 강수 상태, 예보
- **한국환경공단 에어코리아 API** — 통합대기환경지수(CAI)
- **기상청 자외선지수 API** — UV Index
- **카카오 로컬 API** — 장소명 → 위경도 변환

## 아키텍처

```
사용자 브라우저
    ↓
babyweather.jhwan.dev
    ↓
Nginx
    ├── 정적 HTML/CSS/JS 서빙
    └── /api 요청을 FastAPI로 프록시
            ↓
        FastAPI 서비스 레이어
            ├── 카카오 로컬 API
            ├── 기상청 API
            ├── 에어코리아 API
            └── 인메모리 캐시
```

초기에는 프론트엔드 파일을 서버 디렉터리에 직접 두고 서빙했지만, 운영하면서 Nginx와 정적 파일을 Docker 이미지에 함께 넣는 방식으로 바꿨다. 덕분에 로컬에서 검증한 프론트엔드가 그대로 이미지로 배포되고, 수동 복사 없이 업데이트할 수 있게 됐다.

## 점수 계산 로직

총 100점 만점. 각 항목별로 공신력 있는 기준을 적용했다.

| 항목 | 만점 | 기준 |
|------|------|------|
| 기온 | 30점 | 기상청 폭염·한파 특보 기준 |
| 강수확률 | 30점 | 기상청 단기예보 강수확률 |
| 자외선 | 25점 | WHO·질병관리청 UV Index 5단계 |
| 미세먼지 | 15점 | 환경부 통합대기환경지수(CAI) |

예를 들어 기온은 10~28°C면 만점(30점), 33~38°C면 9점, 38°C 초과면 0점이다.

현재 시각에 가까운 조회는 기상청 초단기실황을 사용하고, 가까운 미래는 초단기예보, 더 먼 미래는 단기예보를 사용하도록 분기했다. 사용자는 단순히 시간을 고르지만, 내부에서는 데이터 범위와 응답 크기를 고려해 필요한 API만 호출한다.

## 운영하면서 해결한 문제

### 1. 측정소 탐색 속도 문제

처음엔 에어코리아 API를 직접 호출해서 가장 가까운 측정소를 찾았다. TM 좌표 변환 → 근접 측정소 조회 순서인데, 이 과정에서 API 응답이 느려 카카오 오픈빌더 타임아웃(5초)에 걸렸다.

해결책은 에어코리아에서 전국 673개 측정소 목록(위경도 포함)을 미리 받아서 `stations.json`으로 로컬에 저장하고, 요청마다 직선거리 계산으로 가장 가까운 측정소를 찾는 방식으로 바꿨다. API 호출 한 번이 줄어들면서 응답 속도가 크게 개선됐다.

### 2. 카카오 오픈빌더 발화 매칭 한계와 웹앱 전환

처음엔 카카오톡 챗봇으로 시작했다. 근데 오픈빌더는 사전에 등록된 발화 패턴과 일치해야 응답하는 구조라, "양양공항", "퇴계우성아파트" 같은 자유 지명 입력을 커버하기가 구조적으로 불가능했다.

결국 웹앱으로 방향을 전환했다. 오히려 더 나은 선택이었다. 사용자가 어떤 지명이든 입력할 수 있고, 카카오 API가 알아서 위경도로 변환해준다.

### 3. 공공데이터 API 첫 응답 지연

기상청과 에어코리아 API가 처음 호출 시 응답이 느린 경우가 있었다. 캐시가 없는 새로운 지역 첫 조회 시 3개 API를 `asyncio.gather`로 병렬 호출해도 10초 이상 걸리는 경우가 생겼다.

백엔드에서는 서로 의존성이 없는 외부 API 호출을 병렬화했고, 결과는 원본 JSON 전체가 아니라 필요한 값만 정규화해서 메모리에 저장했다. 같은 요청이 동시에 몰릴 때는 같은 외부 API를 여러 번 때리지 않도록 Single-flight Lock 구조를 넣었다.

프론트에서는 요청이 오래 걸릴 때 자동 재시도와 요청 취소 처리를 넣었다. 이전 요청보다 늦게 도착한 응답이 화면을 덮어쓰지 않도록 요청 순번도 관리한다.

### 4. 기상청 발표 시각과 데이터 지연

기상청 API는 발표 시각과 실제 데이터 생성 시각 사이에 지연이 있다. 예를 들어 특정 기준시각의 예보가 API에 바로 올라오지 않는 경우가 있다. 그래서 단순히 현재 시간만 보고 최신 기준시각을 고르면 빈 응답이 나올 수 있었다.

이를 해결하기 위해 발표 지연을 고려한 버퍼를 두고, 조회 시점에 따라 실황, 초단기예보, 단기예보를 나누도록 바꿨다. API가 비어 있거나 일시적으로 실패해도 화면이 깨지지 않도록 fallback 데이터와 기준시각 보정도 함께 처리했다.

### 5. 대기질 측정 시간 유실 버그

에어코리아 응답에서 통합대기환경지수 값이 비어 있거나 `-`로 들어오는 경우가 있었다. 처음에는 이 값을 기준으로 전체 대기질 데이터를 fallback 처리했는데, 그 과정에서 정상적으로 존재하던 측정 시각까지 사라지는 문제가 생겼다.

이후 점수 계산용 값과 측정 시각을 분리해서 파싱하도록 바꿨다. 점수는 fallback을 타더라도, 데이터에 측정 시각이 있으면 끝까지 보존하도록 고쳤다. 작은 버그였지만, 사용자에게 데이터가 언제 측정된 것인지 보여주는 신뢰성 측면에서는 중요한 수정이었다.

### 6. 라즈베리파이 운영과 배포 자동화

운영 환경은 라즈베리파이 4B다. 개발 환경과 운영 환경의 CPU 아키텍처가 다르기 때문에 Docker 이미지를 멀티 아키텍처로 빌드해야 했다.

GitHub Actions에서 API 이미지와 Web 이미지를 각각 빌드하고 Docker Hub에 푸시하도록 구성했다. 운영 서버에서는 주기적으로 새 이미지를 확인하고, 변경이 있을 때만 컨테이너를 갱신한다. 이 구조 덕분에 로컬에서 검증한 뒤 GitHub에 push하면 프론트엔드와 백엔드가 함께 배포된다.

## 배포와 운영 자동화

운영하면서 중요하게 본 것은 "한 번 고친 것이 다음 배포에서도 그대로 재현되는가"였다.

- 프론트엔드 정적 파일을 Nginx 이미지에 포함
- FastAPI와 Web 이미지를 분리
- GitHub Actions로 멀티 아키텍처 이미지 빌드
- Docker Compose로 서비스 갱신
- GitHub Repository Docs로 릴리즈, 운영 결정, 장애 대응 기록 관리

운영 기록의 원장은 저장소 안에 두기로 했다. `README`, `CHANGELOG`, `ROADMAP`, `docs/decisions`, `docs/operations`, `docs/incidents`를 나눠서 릴리즈와 운영 결정을 코드와 함께 버전 관리한다.

실제 사용자에게 필요한 내용은 서비스의 업데이트 내역과 피드백 페이지에 짧게 노출하고, 기술적으로 공개 가능한 운영 경험은 이 글처럼 `jhwan.dev`의 케이스 스터디로 정리한다.

## 결과

[babyweather.jhwan.dev](https://babyweather.jhwan.dev) 에서 직접 써볼 수 있다.

장소를 검색하면 외출 지수 점수와 함께 기온·강수·자외선·미세먼지 세부 점수, 그리고 각 점수의 기준 출처를 확인할 수 있다. PWA로 만들어서 홈 화면에 앱처럼 설치도 된다.

이 프로젝트를 통해 단순 기능 구현보다 실제 운영에서 더 중요한 것들을 배웠다. 외부 API는 언제든 느리거나 비어 있을 수 있고, 작은 UI 문구 하나도 데이터 신뢰성과 연결된다. 배포 방식이 불안정하면 좋은 기능도 사용자에게 안정적으로 전달되지 않는다.
